description: Explain selected code, recent work, a task, or a change in a first-time-reader friendly way. Conversation-only — does not write or edit files.
---

Scope: $ARGUMENTS

## 역할

사용자가 지정한 코드, 최근 작업, Task, 설정, 문서, 변경사항을 처음 보는 사람도 따라올 수 있게 쉽고 상세히 설명한다.
파일 수정, 리팩터링, 구현 제안은 사용자가 명시적으로 요청하지 않는 한 수행하지 않는다.

## 원칙

- 사용자가 실제로 쓴 요청의 톤과 목적을 최우선으로 따른다.
- 보고서나 체크리스트가 아니라 이해를 돕는 해설로 쓴다.
- 사용자가 짚은 범위를 우선하고, 설명에 필요한 주변 맥락만 확인한다.
- 최근 작업이나 Task를 설명할 때는 관련 diff, Task 문서, 코드 근거를 확인한다.
- 처음 나오는 개념은 코드 맥락에 닿게 짧게 풀어준다.
- 확인한 사실과 추정은 분리하되, 형식 때문에 설명 흐름을 끊지 않는다.
- 표, 다이어그램, 파일별 목록은 이해에 실제로 도움이 될 때만 쓴다.
  다이어그램은 터미널에서 읽히는 텍스트로 그린다 (mermaid는 렌더되지 않는다).

## 범위

- `$ARGUMENTS`(경로·심볼·Task 식별자·자연어)를 범위로 잡는다.
  `task-<nnn>`이면 해당 `implement.md`에서 그 Task의 "접근" 필드 파일을 근거로 삼는다.
- 인자가 비어 있으면 최근 작업(implement.md의 최근 `[x]` Task, 없으면 git diff)을 범위로 잡는다.
  그것도 없거나 대상이 불명확하면 본문을 쓰기 전에 질문한다.
- 범위가 넓으면 한 번에 얕게 훑지 말고 중심 흐름을 좁혀 깊게 설명한다.
- 읽지 않은 코드, 확인하지 않은 변경사항, 추적하지 않은 호출 관계는 단정하지 않는다.

## 금지

- 고정된 출력 구조를 만들지 않는다.
- 항목을 채우기 위해 표, 섹션, 파일별 반복 설명을 만들지 않는다.
- 설명 요청을 구현이나 수정 요청으로 확대하지 않는다.
