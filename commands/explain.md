> 사용 시점: 코드 단위 의도(주석)는 IDE에서 직접 읽고, 흐름·존재 이유·다이어그램을 처음 보는 사람 눈높이의 해설로 풀어 보고 싶을 때.

Scope: $ARGUMENTS

## 역할
- 대상 코드의 흐름·존재 이유·다이어그램을 대화 안에서 풀어 보여준다.
- 자료는 응답에만 존재한다. 파일을 만들거나 수정하지 않는다.
- 코드 단위 의도는 본 커맨드의 범위가 아니다. 그건 코드 옆 주석으로 남기며, 주석 작성 기준은 `skills/implement/SKILL.md`가 소유한다.

## 독자 기준
- 이 코드를 처음 보는 사람이 따라올 수 있게 쓴다. 이미 아는 사람의 복기에도 같은 글이 통한다.
- 등장하는 타입·개념·용어는 처음 나올 때 코드에 닿는 한두 문장으로 소개하고 진행한다.
- 서술 순서는 파일 구조가 아니라 읽는 사람의 이해 순서를 따른다.

## 전제 조건
- 인용한 모든 `file:line`은 실제로 읽은 위치여야 한다 (CLAUDE.md §응답의 일반 룰을 flow 출력에 한정해 강조).

## 인자 처리

### 인자가 있을 때
`$ARGUMENTS`를 자연어 범위 지정으로 해석한다.

- 패키지 경로: 그 패키지가 참여하는 흐름.
- 파일 경로: 그 파일이 참여하는 흐름과 외부 호출자 관계.
- 심볼 이름: 진입점·종착점·중간 노드 중 어디로 둘지 판단해 흐름을 그린다.
- Task 식별자(`task-<nnn>`, feature-dir 병기 가능): 해당 `implement.md`에서 그 Task의 "접근" 필드 파일을 범위로 잡는다.
  feature-dir이 없으면 `features/` 아래에서 해당 Task를 가진 디렉토리를 찾고, 복수면 질문으로 해소한다.
- 자유 자연어: 의미를 추론해 위 갈래 중 하나로 환원한다. 모호하면 본문을 쓰기 전에 질문으로 해소한다.

### 인자가 비어 있을 때
"최근 구현된 부분의 흐름"을 기본 범위로 잡는다.

1. `features/<feature-dir>/implement.md` 가 있으면 가장 최근에 `[x]`로 전환된 Task 1~3개의 "접근" 필드에 언급된 파일을 후보로 잡는다.
2. 1이 없으면 `git status`·`git diff`로 워킹 디렉토리 변경분을 후보로 잡는다.
3. 1·2 모두 비면 커밋 히스토리로 추정하지 않고 사용자에게 범위를 묻는다.

후보가 잡혔으면 본문을 쓰기 전에 "이 범위를 풀겠다"를 한 줄로 먼저 보고한다.

## 작성 절차

1. 대상 파일 전체를 읽는다.
   - 후보 파일 수집은 Glob, 심볼·호출자 위치 탐색은 Grep.
2. 같은 패키지의 test 파일이 있으면 보조 자료로 읽는다. 흐름의 진입점과 계약을 정확히 보여주는 경우가 많다.
3. 호출 사슬은 1단계만 따라간다. 그 이상은 다음 회차 후보로 미룬다.
4. 본문을 작성한다 (§출력 원칙 참조).

## 출력 원칙

- 진입점에서 시작해 호출 순서를 따라가는 하나의 서사로 쓴다. 파일 경계도, §해설 품질 기준의 질문들도
  절 제목으로 세우지 않고 서술 안에서 넘고 답한다.
- 텍스트 다이어그램을 서사의 뼈대로 함께 둔다 (§플로우 다이어그램 규칙).
- 코드·테스트에서 확인되지 않는 실패 모드는 추측해서 넣지 않으며, 직접 확인하지 못한 분기를 다뤘다면
  그 자리에서 명시한다.
- 마무리는 "다음에 보면 좋을 후보 2~3개"만 둔다. 각 한 줄로 "왜 다음인지"를 붙이고, 요약 표나 재열람 목록은 만들지 않는다.

## 플로우 다이어그램 규칙

- 다이어그램은 터미널에서 그대로 읽히도록 코드블록 안 텍스트(ASCII·화살표)로 그린다. mermaid는 렌더되지 않으니 쓰지 않는다.
- 다이어그램은 서사의 지도이지 서사의 복제가 아니다. 흐름 전체를 한 그림에 담지 않고,
  그림 하나는 한 가지(정상 경로의 뼈대, 한 결정 지점의 갈래 등)만 보여준다.
- 노드 이름은 실제 메소드명·타입명을 그대로 쓴다. 다만 `file:line`·부가 설명은 다이어그램이 아니라 해설 본문에 둔다.
- 위에서 아래로 한 단계씩 내려가는 단순한 형태를 기본으로 하고, 열 분할·박스·중첩은 그것 없이는 이해가 안 될 때만 쓴다.
- 두세 문장 해설로 충분한 구간은 다이어그램 없이 푼다.

## 해설 품질 기준

- 처음 보는 독자가 다 읽고 나서 다음에 답할 수 있으면 충분히 담긴 것이다:
  이 흐름은 왜 존재하는가(어떤 시스템 보장을 떠받치는가) / 어디서 시작하는가 /
  어디서 갈리고 다른 선택은 무엇이었는가 / 정상 경로와 같은 깊이에서, 어디서 어떻게 깨지는가 /
  (범위가 최근 변경분이면) 무엇이 어떻게 왜 바뀌었는가.

## 분량 가이드

- 흐름 해설(다이어그램·진입점·결정 지점·실패 경로)은 줄이지 않는다.
- 흐름이 한 응답에 다 풀기 어려운 분량이면 "범위를 X·Y까지로 좁히겠다"를 먼저 보고하고 좁혀서 깊이를 유지한다. 깊이를 줄여 범위를 욱여넣지 않는다.

## 금지

- 일반론·교과서 인용을 코드와의 연결 없이 늘어놓지 않는다.

## 후속 행동
- 사용자가 후보 중 하나를 고르거나 직접 범위를 지정해 다시 `/explain`을 부르는 것을 기대한다. 자동으로 이어 실행하지 않는다.
