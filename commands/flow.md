> 사용 시점: 코드 단위 의도(주석)는 IDE에서 직접 읽고, 흐름·존재 이유·다이어그램만 대화에서 풀어 보고 싶을 때.

Scope: $ARGUMENTS

## 역할
- 대상 코드의 흐름·존재 이유·다이어그램을 대화 안에서 풀어 보여준다.
- 자료는 응답에만 존재한다. 파일을 만들거나 수정하지 않는다.
- 코드 단위 의도는 본 커맨드의 범위가 아니다. 그건 코드 옆 주석으로 남기며, 주석 작성 기준은 `skills/implement/SKILL.md`가 소유한다.

## 전제 조건
- 인용한 모든 `file:line`은 실제로 읽은 위치여야 한다 (CLAUDE.md §응답의 일반 룰을 flow 출력에 한정해 강조).

## 인자 처리

### 인자가 있을 때
`$ARGUMENTS`를 자연어 범위 지정으로 해석한다.

- 패키지 경로: 그 패키지가 참여하는 흐름.
- 파일 경로: 그 파일이 참여하는 흐름과 외부 호출자 관계.
- 심볼 이름: 진입점·종착점·중간 노드 중 어디로 둘지 판단해 흐름을 그린다.
- 자유 자연어: 의미를 추론해 위 갈래 중 하나로 환원한다. 모호하면 본문을 쓰기 전에 질문으로 해소한다.

### 인자가 비어 있을 때
"최근 구현된 부분의 흐름"을 기본 범위로 잡는다.

1. `docs/<feature-dir>/implement.md` 가 있으면 가장 최근에 `[x]`로 전환된 Task 1~3개의 "접근" 필드에 언급된 파일을 후보로 잡는다.
2. 1이 없으면 `git status`·`git diff`로 워킹 디렉토리 변경분을 후보로 잡는다.
3. 1·2 모두 비면 커밋 히스토리로 추정하지 않고 사용자에게 범위를 묻는다.

후보가 잡혔으면 본문을 쓰기 전에 "이 범위를 풀겠다"를 한 줄로 먼저 보고한다.

## 작성 절차

1. 대상 파일 전체를 읽는다.
   - 후보 파일 수집은 Glob, 심볼·호출자 위치 탐색은 Grep.
2. 같은 패키지의 test 파일이 있으면 보조 자료로 읽는다. 흐름의 진입점과 계약을 정확히 보여주는 경우가 많다.
3. 호출 사슬은 1단계만 따라간다. 그 이상은 다음 회차 후보로 미룬다.
4. 본문을 작성한다 (§출력 구조 참조).

## 출력 구조

응답 본문은 이 순서를 따른다.

````markdown
# <범위 이름> — 흐름 해설

> (선택) 이 회차의 범위·전제 한 문단.

## 1. <첫 파일/패키지 이름>

### 1-1. 존재 이유
1~3문단. 시스템 안에서의 역할, 이 자리가 사라지면 어디서 문제가 터지는지. "무엇을 한다"가 아니라 "왜 있어야 하는가".

### 1-2. 흐름
다이어그램 + 텍스트 해설. 다이어그램 선택은 §플로우 다이어그램 규칙 참조.

해설:
- 진입점을 한 줄로 명시.
- 결정 지점·분기·동시성 구간을 풀어 적는다.
- 정상 경로를 먼저 끝까지 푼다.
- 실패·재시도·timeout 경로도 정상 경로와 동등한 깊이로 푼다 — 어디서 깨지는가, 무엇을 재시도하는가, 어떤 timeout이 어디에 걸리는가. 코드·테스트에서 확인되지 않는 실패 모드는 추측해서 넣지 않는다.

### 1-3. 외부 사용 관계
호출자, 호출 시점, 현재 사용 여부, 앞으로 이동할 수 있는 책임.

## 2. <두 번째 파일/패키지 이름>
… (1과 같은 구조)

## 회차 마무리

| 파일 | 한 줄 정리 | 다른 곳에서 의존하는 방식 |
|---|---|---|
| … | … | … |

다시 들여다볼 파일:
- … (이번 흐름에서 본 것 중 한 번 더 봐야 이해가 굳을 곳)

인지하고 있어야 할 미검증 경로:
- … (diff·테스트로 직접 확인되지 않은 분기·경로만. 가설적 우려는 두지 않는다)

다음에 보면 좋을 후보:
- … (2~3개, 각 한 줄로 "왜 다음인지" 이유 포함)
````

## 플로우 다이어그램 규칙

- 다이어그램은 터미널에서 그대로 읽히도록 코드블록 안 텍스트(ASCII·화살표)로 그린다. mermaid는 렌더되지 않으니 쓰지 않는다.
- 흐름 절은 텍스트 다이어그램과 텍스트 해설을 함께 둔다. 한쪽만 두지 않는다.
- 표현 방식은 흐름 성격에 맞춰 가독성 좋은 텍스트 형태를 택한다. 특정 표기법을 강제하지 않는다.
- 노드 이름은 실제 메소드명·타입명을 그대로 쓴다. 추상화하지 않는다.
- 다이어그램이 한 화면을 넘길 만큼 커지면 나눈다.

## 해설 품질 기준

- "이 흐름이 어떤 시스템 보장을 떠받치는가"가 드러나는가. 진입점·결정 지점·실패 경로를 푼 뒤에도 이 질문에 답이 없다면 §1-1 존재 이유를 보강한다.

## 분량 가이드

- 이 출력은 빠른 복기를 위한 자료다. 흐름 절(다이어그램·진입점·결정 지점·실패 경로)은 줄이지 않는다.
- 흐름이 한 응답에 다 풀기 어려운 분량이면 "범위를 X·Y까지로 좁히겠다"를 먼저 보고하고 좁혀서 깊이를 유지한다. 깊이를 줄여 범위를 욱여넣지 않는다.

## 금지

- 일반론·교과서 인용을 코드와의 연결 없이 늘어놓지 않는다.

## 후속 행동
- 사용자가 후보 중 하나를 고르거나 직접 범위를 지정해 다시 `/flow`를 부르는 것을 기대한다. 자동으로 이어 실행하지 않는다.

## 핵심 질문
