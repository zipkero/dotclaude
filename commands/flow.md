Pairs with code-level intent expressed as comments. Use when the user wants to understand how a package/file/symbol fits into the overall flow, how data and control move through it, or asks to explain recent work as a flow walkthrough. Never write files unless explicitly asked.

> 사용 시점: 코드 단위 의도(주석)는 IDE에서 직접 읽고, 흐름·존재 이유·다이어그램만 대화에서 풀어 보고 싶을 때.

Scope: $ARGUMENTS

## 역할
- 대상 코드의 흐름·존재 이유·다이어그램을 대화 안에서 풀어 보여준다.
- 자료는 응답에만 존재한다. 파일을 만들거나 수정하지 않는다.
- 코드 단위 의도는 본 커맨드의 범위가 아니다. 그건 코드 옆 주석으로 남기며, 주석 작성 기준은 `skills/implement/SKILL.md`가 소유한다.

## 전제 조건
- 사용자가 코드 수정을 명시 요청하기 전에는 어떤 파일도 건드리지 않는다.
- 인용한 모든 `file:line`은 실제로 읽은 위치여야 한다. 안 읽은 파일·검증 안 된 외부 자료는 단언하지 않는다.

## 인자 처리

### 인자가 있을 때
`$ARGUMENTS`를 자연어 범위 지정으로 해석한다.

- 패키지 경로: 그 패키지가 참여하는 흐름.
- 파일 경로: 그 파일이 참여하는 흐름과 외부 호출자 관계.
- 심볼 이름: 진입점·종착점·중간 노드 중 어디로 둘지 판단해 흐름을 그린다.
- 회차 번호: 기존 회차 목록이 있으면 지정 회차에 들어갈 범위를 직접 정한다.
- 자유 자연어: 의미를 추론해 위 갈래 중 하나로 환원한다. 모호하면 본문을 쓰기 전에 한 번 질문한다.

### 인자가 비어 있을 때
"최근 구현된 부분의 흐름"을 기본 범위로 잡는다.

1. `docs/<feature-dir>/implement.md` 가 있으면 가장 최근에 `[x]`로 전환된 Task 1~3개의 "접근" 필드에 언급된 파일을 후보로 잡는다.
2. 1이 없으면 `git status`·`git diff`로 워킹 디렉토리 변경분을 먼저 후보로 잡고, 그것도 비면 `git log --name-only -n 5`로 최근 커밋의 변경 파일을 후보로 잡는다.
3. 1·2 모두 비면 진행하지 않고 사용자에게 범위를 묻는다.

후보가 잡혔으면 본문을 쓰기 전에 "이 범위를 풀겠다"를 한 줄로 먼저 보고한다.

## 작성 절차

1. 대상 파일 전체를 Read로 읽는다. 후보 파일 수집은 Glob, 심볼·호출자 위치 탐색은 Grep. Read 한계(2000줄) 초과 시 청크로 나눠 모두 읽는다.
2. 같은 패키지의 test 파일이 있으면 보조 자료로 읽는다. 흐름의 진입점과 계약을 정확히 보여주는 경우가 많다.
3. 호출 사슬은 1단계만 따라간다. 그 이상은 "다음 회차 후보"로 미룬다.
4. 본문을 작성한다 (§출력 구조 참조).
5. 마지막에 다음에 보면 좋을 후보 2~3개를 짧게 제안한다.

## 출력 구조

응답 본문은 이 순서를 따른다.

````markdown
# <범위 이름> — 흐름 해설

> (선택) 이 회차의 범위·전제 한 문단.

## 1. <첫 파일/패키지 이름>

### 1-1. 존재 이유
1~3문단. 시스템 안에서의 역할, 이 자리가 사라지면 어디서 문제가 터지는지. "무엇을 한다"가 아니라 "왜 있어야 하는가".

### 1-2. 흐름
다이어그램 + 텍스트 해설. 다이어그램 선택은 §플로우 다이어그램 규칙 참조.

해설:
- 진입점을 한 줄로 명시.
- 결정 지점·분기·동시성 구간을 풀어 적는다.
- 정상 경로를 먼저 끝까지, 에러 경로는 한 줄로 요약.

### 1-3. 외부 사용 관계
호출자, 호출 시점, 현재 사용 여부, 앞으로 이동할 수 있는 책임.

## 2. <두 번째 파일/패키지 이름>
… (1과 같은 구조)

## 테스트가 보장하는 계약

| 테스트 | 보장하는 것 |
| --- | --- |
| `Test...` | ... |

테스트가 없는 범위라면 이 절을 통째로 생략한다.

## 회차 마무리

| 파일 | 한 줄 정리 | 다른 곳에서 의존하는 방식 |
|---|---|---|
| … | … | … |

다음에 보면 좋을 후보:
- … (2~3개, 각 한 줄로 "왜 다음인지" 이유 포함)
````

## 해설 품질 기준

- 진입점과 종착점이 명시되었는가.
- 설명이 필요한 결정 지점·분기·동시성 구간이 다이어그램과 텍스트 양쪽에 모두 드러나는가.
- "이 흐름이 어떤 시스템 보장을 떠받치는가"가 드러나는가.
- 안 읽은 파일에 대해 추측하지 않았는가.

## 플로우 다이어그램 규칙

- 흐름 절은 다이어그램과 텍스트 해설을 함께 둔다. 다이어그램만 두거나 텍스트만 두지 않는다.
- 흐름 성격에 맞춰 고른다:
  - 호출·메시지 순서: mermaid `sequenceDiagram`
  - 분기·결정 로직: mermaid `flowchart`
  - 상태 전이·라이프사이클: mermaid `stateDiagram-v2`
  - 단선형 호출 사슬: 텍스트 화살표(`→`)
  - 동시성(goroutine·worker pool 등)도 sequenceDiagram에 lane을 나눠 표현한다.
- 노드 이름은 실제 메소드명·타입명을 그대로 쓴다. 추상화하지 않는다.
- 다이어그램 1개가 한 화면을 넘기면(노드 15개 이상) 두 개로 분할한다.

## 분량 가이드

- 흐름이 한 응답에 다 풀기 어려운 분량이면 "범위를 X·Y까지로 좁히겠다"를 먼저 보고하고 좁힌다.
- 진입점·결정 지점·종착점이 한 줄씩으로 끝났다면 깊이 부족 신호다. 결정 지점의 "왜"와 시스템 보장을 보강한다.

## 금지

- 어떤 파일도 작성·수정·삭제하지 않는다.
- 코드 단위 의도(메소드 한 개의 "왜 이 모양인가")는 본 커맨드에서 풀지 않는다. 그건 주석의 역할이다.
- 안 읽은 파일을 인용하지 않는다.
- 일반론·교과서 인용을 코드와의 연결 없이 늘어놓지 않는다.

## 후속 행동
- 본문 끝에 "다음에 보면 좋을 후보"를 2~3개 둔다.
- 사용자가 후보 중 하나를 고르거나 직접 범위를 지정해 다시 `/flow`를 부르는 것을 기대한다. 자동으로 이어 실행하지 않는다.

## 핵심 질문
> 이 흐름은 어디서 시작해 어디서 끝나는가? 결정 지점마다 다른 선택은 무엇이었는가? 이 흐름이 시스템의 어떤 보장을 어디서 떠받치고 있는가?
