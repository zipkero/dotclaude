---
description: Launch N independent agents that each analyze the SAME question with an identical prompt, then main cross-verifies their reports and returns a consensus/disagreement summary. Read-only — no files are written. Use when the user wants a rigorous, high-confidence analysis of a code/behavior question and explicitly wants multiple agents cross-checked (e.g. "3개 에이전트로 각각 분석해서 교차검증해줘").
---

> 사용 시점: 결론의 신뢰도가 중요해 여러 에이전트에게 같은 질문을 독립적으로 풀게 하고, 메인이 그 결과를 교차검증해 합의·불일치를 가려내고 싶을 때. 단일 관점으로 충분하면 자연어 `analyze`를 쓴다.

Scope: $ARGUMENTS

## 역할
- 하나의 분석 질문을 **N개 에이전트가 동일한 프롬프트로 각각 독립 분석**하게 하고, 메인이 반환 보고를 **교차검증**해 하나의 결론으로 합친다.
- 산출은 대화 응답에만 존재한다. 에이전트도 메인도 **파일을 만들거나 수정하지 않는다** (읽기 전용 분석).
- 구현 요청으로 해석하지 않는다. 분석·검토·설명·비교만 한다. 구현이 필요해 보이면 범위와 다음 단계만 보고한다.

## 인자 처리
`$ARGUMENTS` 파싱:
- 맨 앞 토큰이 정수면 그 값을 **에이전트 수 N**으로 쓰고, 나머지를 분석 대상으로 본다. (예: `/cross-analyze 5 <질문>`)
- 정수가 없으면 **N=3**(기본), 전체를 분석 대상으로 본다.
- 대상이 비어 있으면 직전 대화 맥락에서 "가장 최근에 논의된 미해결 분석 질문"을 후보로 잡고, 본문 착수 전에 "이 질문을 N개로 교차분석하겠다"를 한 줄로 먼저 보고한다. 후보가 불명확하면 질문으로 해소한다.

## 절차

1. **분석 브리프 1개 작성** — 메인이 대상 질문을 다음을 포함한 단일 프롬프트로 정리한다. 이 프롬프트는 **모든 에이전트에게 토씨까지 동일하게** 전달한다 (수렴 검증이 목적이므로 각도를 다르게 주지 않는다).
   - 분석 질문(무엇을 답해야 하는가)과 반드시 답할 하위 질문 목록
   - 읽어야 할 핵심 파일/심볼 경로 힌트(아는 범위에서). 절대경로로.
   - 산출 규칙(아래 §에이전트 프롬프트 고정 규칙)
2. **N개 에이전트 병렬 실행** — `general-purpose` 서브에이전트 N개를 **한 메시지에서 동시에** 띄운다(각각 별도 Agent 호출). 동일 프롬프트를 넣는다.
3. **완료 대기** — 전부 완료될 때까지 기다린다. 중간에 메인이 같은 파일을 직접 분석해 작업을 중복하지 않는다.
4. **교차검증** — 반환된 N개 보고를 §교차검증 규칙으로 대조해 하나로 합친다. 필요하면 메인이 결정적 근거 1~2곳만 직접 열어 확인한다(전면 재분석 금지).
5. **보고** — §출력 구조로 사용자에게 보고한다.

## 에이전트 프롬프트 고정 규칙
모든 에이전트 프롬프트에 아래를 명시한다(동일 문구):
- 코드를 직접 읽고 **모든 주장에 `file:line` 근거**를 붙인다.
- **확인된 사실과 추정을 반드시 분리**해 보고한다.
- **파일을 수정하지 않는다** (읽기 전용).
- 결론을 한 줄로 먼저 제시하고, 그 뒤 근거를 단다.
- 사용자 결정이 필요한 지점을 발견하면 코드·문서를 건드리지 않고 그 사실을 보고에 남긴다.
- 산출 형식: ①한 줄 결론 ②근거(file:line 포함) ③확인 사실 vs 추정 구분.

## 교차검증 규칙
- **합의**: N개 중 다수가 같은 근거(가능하면 같은 file:line)로 도달한 결론. "N/N" 또는 "몇 개가 동의했는지"를 표기한다.
- **불일치**: 결론이 갈린 지점은 숨기지 않는다. 누가 무엇을 다르게 말했는지, 어느 쪽 근거가 더 강한지(직접 확인한 line 기준)를 적는다. 필요 시 메인이 해당 line만 열어 판정한다.
- **단독 발견**: 한 에이전트만 짚은 유효한 발견도 근거가 확인되면 채택하고 "단독 발견"으로 표기한다.
- **사실/추정 통합**: 에이전트별 사실/추정을 합쳐, 최종적으로 확인된 사실과 남은 추정을 다시 가른다. 한 에이전트가 "추정"이라 한 걸 다른 에이전트가 line으로 확인했으면 "사실"로 승격한다.
- 에이전트가 지정 파일을 못 찾았거나 프록시로 대체했으면 그 한계를 보고에 명시한다(결론 신뢰도에 영향).

## 출력 구조
````markdown
# <대상> — N개 에이전트 교차검증

## 결론 (한 줄 + 신뢰도)
- 핵심 답 + "N/N 합의" 또는 "불일치 있음" 표기.

## 합의 사항
- 근거 line과 함께. (동의한 에이전트 수 표기)

## 불일치 및 해소
- 갈린 지점 / 각 주장 / 어느 쪽이 맞는지 + 판정 근거(직접 확인한 line).
- 불일치가 없으면 "없음"이라고 적는다.

## 단독 발견 (있으면)
- 한 에이전트만 짚었으나 근거 확인된 것.

## 확인된 사실 vs 남은 추정
- 사실: … (file:line)
- 추정/미확인: … (왜 확정 못 했는지, 확정하려면 무엇이 필요한지)

## 남은 질문 / 다음 단계
- 사용자 결정이 필요한 지점, 또는 추가로 열어봐야 할 대상.
````

## 금지
- 파일 생성·수정·삭제. (읽기 전용)
- 에이전트마다 다른 각도를 주는 것. 이 커맨드는 수렴 검증이 목적이라 프롬프트를 동일하게 둔다. (각도를 나누고 싶으면 이 커맨드가 아니라 개별 지시로 처리)
- 근거 line 없이 단정하기. 확인 못 한 건 추정으로 분류한다.
- 분석을 구현으로 확장하기.

## 후속 행동
- 사용자가 결론을 받고 추가 분석·구현 여부를 지시하기를 기대한다. 자동으로 이어가지 않는다.

## 핵심 질문
> 같은 질문을 독립적으로 풀었을 때 결론이 수렴하는가? 갈린 지점은 어디이고, 근거로 어느 쪽이 옳은가? 확정된 사실과 아직 추정인 것은 각각 무엇인가?
